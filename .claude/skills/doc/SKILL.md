# doc - 팀 문서 작성 Skill

## 개요

프로젝트의 역할별 문서를 표준화된 구조와 포맷으로 작성/관리한다.
문서 루트 경로: `docs/` (프로젝트 루트 기준 상대 경로)

## 사용법

사용자가 `/doc` 뒤에 명령을 입력한다.

### 명령어

| 명령 | 설명 | 예시 |
|------|------|------|
| `init <Name>` | 문서 디렉토리 초기화 | `/doc init Dev_Client` |
| `troubleshoot <Name> <IssueName>` | 트러블슈팅 문서 작성 | `/doc troubleshoot Dev_Client iOS빌드실패` |
| `todo <Name> <TodoName>` | Todo 문서 작성 | `/doc todo Dev_Client 로그인모듈리팩토링` |
| `log <Name>` | 작업일지 항목 추가 | `/doc log Dev_Client` |
| `mission <Name>` | 임무(역할) 문서 작성/수정 | `/doc mission Dev_Client` |
| `design <Name> <DesignName>` | 시스템기획 문서 작성 | `/doc design Dev_Client 인증모듈` |
| `knowledge <Name> <Title>` | 지식 문서 작성 | `/doc knowledge Claude_Code_Engineer Prompt_Caching` |
| `new <Name> <Category> <DocName>` | 자유 카테고리 문서 작성 | `/doc new Dev_Client 시스템설계 인증모듈` |
| `list <Name>` | 문서 목록 조회 | `/doc list Dev_Client` |

명령어 없이 `/doc`만 입력하면 사용자에게 무엇을 할지 질문한다.

---

## 실행 절차

### 1. init - 문서 디렉토리 초기화

지정된 Name으로 아래 디렉토리 구조를 생성한다.

```
docs/
  {Name}/
    Mission.md                    ← 임무(역할) 정의
    TroubleShooting/              ← 트러블슈팅 문서
    Todo/                         ← Todo 문서
    SystemDesign/                 ← 시스템기획 문서
    Knowledge/                    ← 지식 문서 (조사/학습한 정보)
    WorkLog/                      ← 작업 일지
      {YYYY-MM}.md                 ← 월별 작업일지
```

추가로 역할 성격에 따라 필요한 하위 폴더를 사용자에게 확인한 뒤 생성한다.
예시:
- Dev 계열: `DevEnvironment/`, `Architecture/`, `ETC/`
- CICD 계열: `Pipeline/`, `BuildConfig/`, `ETC/`
- QA 계열: `TestCases/`, `TestResults/`, `ETC/`

Mission.md 초기 내용은 사용자에게 역할/책임을 질문하여 작성한다.

디렉토리 생성 시 bash brace expansion(`{A,B,C}`)을 사용하지 않는다. 폴더마다 개별 `mkdir -p` 명령을 사용한다.

### 2. troubleshoot - 트러블슈팅 문서

경로: `docs/{Name}/TroubleShooting/{IssueName}.md`

#### 포맷

```markdown
# {IssueName}

## 발생일
{YYYY-MM-DD}

## 환경
- OS:
- 엔진/프레임워크 버전:
- 관련 빌드/커밋:

## 증상
{어떤 현상이 발생했는지 구체적으로}

## 원인
{근본 원인. 검증 전이면 "추정" 명시}

## 해결 방법
{단계별 해결 절차}

## 예방 방법
{재발 방지를 위한 조치}

## 관련 문서
- {연관 문서 링크/경로}
```

작성 시 규칙:
- 증상, 원인, 해결 방법은 반드시 채운다
- 원인이 확인되지 않았으면 "추정"을 명시한다
- 해결하지 못한 경우 해결 방법에 "미해결 - 시도한 내용:" 을 적고 시도한 것들을 나열한다

### 3. todo - Todo 문서

경로: `docs/{Name}/Todo/{TodoName}.md`

#### 포맷

```markdown
# {TodoName}

## 우선순위
{Critical / High / Medium / Low}

## 기획 의도
{왜 이 작업이 필요한지}

## 작업 목표
{완료 조건이 무엇인지}

## 마감일
{YYYY-MM-DD 또는 미정}

## 예상 작업 시간
{예상 소요 시간. 모르면 "미정"}

## 관련 문서
- {연관 문서 경로/링크}

## 해야 할 작업
- [ ] {작업 항목}

## 진행 중 작업
- [ ] {작업 항목} - {담당자/상태}

## 완료된 작업
- [x] {작업 항목} ({완료일})
```

작성 시 규칙:
- 기존 Todo 문서가 있으면 항목을 업데이트한다 (새로 만들지 않는다)
- 작업이 진행되면 "해야 할 작업"에서 "진행 중 작업"으로 이동
- 작업이 완료되면 "완료된 작업"으로 이동하고 완료일 기록

### 4. log - 작업일지

경로: `docs/{Name}/WorkLog/{YYYY-MM}.md`

월별로 하나의 파일을 사용한다. 기존 파일이 있으면 항목을 추가한다.

#### 포맷

```markdown
# 작업일지 {YYYY-MM}

## {YYYY-MM-DD}

### 수행 작업
- {작업 내용 요약}

### 주요 변경사항
- {변경된 파일/설정/구성}

### 작업 소요시간
{실제 소요된 시간. 모르면 생략}

### 관련 문서
- {연관 문서 경로/링크. 없으면 생략}

### 이슈/메모
- {특이사항, 발견한 문제, 참고사항}
```

작성 시 규칙:
- 너무 세세하게 적지 않는다 (핵심만 간결하게)
- 같은 날짜 항목이 있으면 기존 항목에 추가한다
- 코드 변경의 경우 파일 경로 정도만 기록, 코드 자체는 적지 않는다

### 5. design - 시스템기획 문서

경로: `docs/{Name}/SystemDesign/{DesignName}.md`

#### 포맷

```markdown
# {DesignName}

## 작성일
{YYYY-MM-DD}

## 기획 의도
{왜 이 시스템/기능이 필요한지}

## 목표
{이 시스템이 달성해야 하는 것}

## 동작 방식
{시스템이 어떻게 동작하는지 구체적으로}

## 관련 문서
- {연관 문서 경로/링크}

## 작업 내용
- {구현/변경해야 할 항목들}
```

작성 시 규칙:
- 기획 의도와 목표를 명확히 구분한다 (왜 vs 무엇)
- 동작 방식은 누가 읽어도 구현할 수 있을 정도로 구체적으로 작성
- 관련 Todo 문서가 있으면 연결한다

### 6. knowledge - 지식 문서

경로: `docs/{Name}/Knowledge/{Title}.md`

조사, 학습, 분석을 통해 알아낸 정보를 축적하는 문서. 나중에 동일한 주제를 다시 조사하지 않도록 지식을 기록한다.

#### 포맷

```markdown
# {Title}

## 작성일
{YYYY-MM-DD}

## 최종 수정일
{YYYY-MM-DD}

## 요약
{핵심 내용을 3줄 이내로}

## 상세 내용
{조사/학습한 내용을 구조적으로 정리}

## 출처
- {공식 문서 URL, 코드 경로, 실험 결과 등 정보의 근거}

## 관련 문서
- {연관 문서 경로/링크. 없으면 생략}
```

작성 시 규칙:
- 출처는 반드시 기재한다 (URL, 코드 경로, 실행 결과 등)
- 공식 문서에서 확인한 내용과 추정/경험 기반 내용을 구분한다
- 기존 문서가 있으면 "최종 수정일"을 갱신하고 내용을 업데이트한다 (새로 만들지 않는다)
- 시간이 지나면 변할 수 있는 정보(버전별 동작 등)는 확인 시점을 명시한다

### 7. mission - 임무(역할) 문서

경로: `docs/{Name}/Mission.md`

#### 포맷

```markdown
# {Name} 임무

## 역할
{담당하는 역할}

## 책임 범위
- {구체적인 책임 항목들}

## 권한/제약
- {할 수 있는 것 / 하면 안 되는 것}

## 주요 작업 흐름
1. {일반적인 작업 순서}

## 사용 도구/환경
- {사용하는 도구, 환경 정보}

## 협업 대상
- {협업하는 대상과 협업 방식}
```

### 8. new - 자유 카테고리 문서

경로: `docs/{Name}/{Category}/{DocName}.md`

Category 폴더가 없으면 생성한다. 문서 포맷은 내용에 맞게 자유롭게 작성하되, 최소한 제목과 작성일은 포함한다.

### 9. list - 문서 목록 조회

디렉토리 하위의 모든 문서를 트리 형태로 표시한다.

---

## 공통 규칙

1. **문서 경로**: 모든 문서는 `docs/{Name}/` 하위에 위치 (프로젝트 루트 기준 상대 경로)
2. **파일명 규칙**: 공백 대신 언더스코어(`_`) 사용, 한글 허용
3. **기존 문서 확인**: 문서를 새로 만들기 전에 동일/유사 문서가 있는지 먼저 확인
4. **작성 후 보고**: 문서 작성/수정 후 경로와 요약을 사용자에게 보고
