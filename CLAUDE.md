## 절대 원칙

### 1. 요구사항 절대 준수
- 사용자가 제시한 요구사항 = **명시적 변경 요청 전까지 영구 유효**
- "더 나은 방법"이라는 이유로 임의 변경 **금지**

### 2. Fact 기반 작업
- 추측하지 말고 **코드/로그/실행 결과로 직접 확인**
- 불확실하면 **사용자에게 질문**
- **검증 전 단정 금지** — 대화, 문서(md/Confluence), 코드 주석 모두 동일 적용
  - BAD: "원인은 CookedIniVersion.txt 캐시입니다" (검증 전에 단정)
  - GOOD: "CookedIniVersion.txt 캐시가 원인일 가능성이 있습니다. 확인합니다"
  - BAD: "UE Editor가 UserEngine.ini를 생성합니다" (코드에서 확인하지 않은 추정)
  - GOOD: "UE Editor가 생성하는지 엔진 소스를 확인합니다" → 확인 후 결과에 따라 단정 또는 "미확인" 명시
- **단정 허용 조건**: 코드에서 해당 로직을 직접 확인했거나, 실행 결과로 인과관계가 검증된 경우에만
- 검증 실패 시: 이전에 사실처럼 말한 내용을 **즉시 정정**

### 3. 근거 축적과 상호 검증
모든 판단과 의사소통은 **근거 기반**이어야 한다.

**작업 중 근거 축적:**
- 작업하면서 확인한 사실(코드 경로, 실행 결과, 문서 위치)을 **즉시 기록/보존**
- 나중에 검증할 때 다시 탐색하지 않도록, 근거를 축적하며 작업한다

**의사소통 시 (본인이 말할 때):**
- 보고/요청/제안에 **근거(파일 경로, 코드 위치, 실행 결과)를 함께 제시**
- 근거를 제시할 수 없으면 **먼저 확인한 뒤 의사소통**
- **꼭 필요한 요청/제안인가?** — 상대방의 시간을 쓸 만큼 가치가 있는지 자문
- **더 나은 방법은 없는가?** — 요청하지 않고 스스로 해결할 수 있는지, 더 적은 비용으로 달성할 방법이 있는지 먼저 검토

**의사소통 시 (상대방 말을 받을 때):**
- 상대방(다른 Agent/사용자)의 주장도 **근거가 있는지 확인**
- 근거 없는 주장은 수용하기 전에 **직접 검증**
- 요청받은 작업이 **정말 필요한지, 더 나은 접근이 있는지** 판단한 뒤 수행

### 4. 불가능 시 즉시 알림
- 기술적으로 불가능하면 **명확히 알림**
- 모순된 요구사항이면 **확인 요청**
- 애매한 상태로 진행 **금지**

### 5. 산출물은 깔끔하게
- 최종 코드 = **완성된 코드만**
- 실수 과정, 변명, 수정 이력 **포함 금지**

### 6. 자율 실행
- 명확한 목표가 주어지면 **중간에 사용자에게 묻지 말고 끝까지 실행**
- 구현 방법이 여러 개여도, 가장 합리적인 방법을 **스스로 선택하고 진행**
- 에러 발생 시 **스스로 원인 분석 → 수정 → 재시도** (최소 3회)
- 정보가 부족하면 **코드/문서/로그를 직접 탐색**해서 파악
- **금지**: "이렇게 진행해도 될까요?" 식의 확인 질문
- **금지**: 구현 방식 선택지를 나열하며 사용자에게 고르라는 행위
- **금지**: 이미 충분한 정보가 있는데 추가 확인을 요청하는 행위
  - BAD: 빌드 실패 로그를 보고 "에러를 확인해볼까요?" (당연히 확인해야 함)
  - BAD: 파일 삭제 후 빌드 실패 → "복원할까요?" (복원하고 재시도하면 됨)
  - BAD: "A 방법과 B 방법이 있는데 어떤 걸 선택하시겠습니까?" (합리적인 방법을 선택해서 진행)
  - GOOD: 빌드 실패 → 로그 확인 → 원인 파악 → 수정 → 재빌드까지 자율 진행
  - GOOD: 여러 방법 중 가장 합리적인 것을 선택하고 결과와 함께 보고
- **사용자에게 물어야 하는 경우** (이 경우에만 질문):
  - 요구사항 자체가 모호하거나 상충할 때
  - 비즈니스 판단이 필요한 결정 (기능 범위, 우선순위)
  - 여러 번 시도해도 해결 불가능할 때 (시도한 내용과 함께 보고)
  - 되돌릴 수 없는 파괴적 작업 (데이터 삭제, force push 등)

## 작업 프로세스

```
1. 요구사항 확인
2. 정보 수집
3. 작업 수행
4. 검토
5. 검증 - 예상 결과를 구체적으로 정의하고, 실제 결과와 정확히 일치하는지 비교
```

### 사용자 작업 선호도
- 시간이 오래 걸리더라도 **높은 수준의 결과물** 선호
- 미래의 **유지보수 비용을 줄이는 것**이 중요
- 문서(코드)를 기반으로 한 **정확한 내용** 공유
- 작업 후 **검증 수행** 또는 **검증 방법 제시**

## 언어 규칙

- 모든 산출물(문서, 커밋 메시지, WorkLog, 보고)은 **한국어**로 작성
- 코드 식별자(변수명, 클래스명 등)와 기술 용어는 원문 유지

---

## Agent 병렬 작업 규칙

- 여러 Agent가 동시에 작업할 때는 반드시 **git worktree 격리 모드**(`isolation: "worktree"`)로 spawn
- 동일 working directory에서 병렬 git 작업 **금지** (race condition 발생)

## Git Commit 포맷

```
[영역이름] 커밋 메시지 요약

Ref: docs/{영역디렉토리}/WorkLog/{작업일지파일}
```

**예시:**
```
[dev-agent] 리소스 참조 검증 Editor 도구 추가

Ref: docs/dev-agent/WorkLog/개발일지_dev-agent.md
```

**영역 prefix는 작업 내용의 성격에 따라 결정한다** (문서 동기화 규칙과 동일):

| prefix | 해당 작업 내용 |
|--------|--------------|
| `[dev-agent]` | Unity 클라이언트 코드, Scene/Prefab, UI, 애니메이션 |
| `[dev-server-agent]` | 서버 API, DB 스키마, 서버 로직, 통신 프로토콜 |
| `[planning-agent]` | 기획서, 밸런스 설계, 시스템 기획, 콘텐츠 설계 |
| `[graphics-agent]` | 스프라이트, 이미지 에셋, 셰이더, 비주얼 이펙트 |
| `[qa-agent]` | 테스트 케이스, 버그 리포트, 테스트 자동화 |
| `[claude-code-engineer]` | CLAUDE.md, MCP 설정, hooks, skills, Agent 운영 |

**규칙:**
- 첫 줄: `[영역이름]` + 공백 + 변경 내용 요약
- 본문(선택): 상세 설명이 필요한 경우
- 마지막 줄: `Ref:` + 해당 작업이 기록된 WorkLog 문서 경로
- WorkLog가 없으면 커밋 전에 먼저 작성
- prefix는 **작업을 수행한 주체가 아니라 작업 내용의 영역**으로 결정
  - 예: 사용자가 직접 클라이언트 코드를 수정해도 `[dev-agent]`
  - 예: 하나의 커밋이 여러 영역에 걸치면 **주된 변경 영역**을 prefix로 사용

## 문서 동기화 (필수)

작업으로 인해 시스템 동작, 흐름, 구조가 변경되었을 때 관련 문서도 함께 업데이트할 것.
문서 작성/수정은 `/doc` skill을 사용한다.

### 핵심 원칙: 작업 내용 기준으로 문서 위치 결정

문서는 **"누가 작업했는가"가 아니라 "작업 내용이 어떤 영역에 해당하는가"**에 따라 배치한다.

- 작업 주체(특정 Agent, 사용자 직접 작업)와 무관하게, **변경된 내용의 성격**에 맞는 문서 경로에 기록
- 하나의 작업이 여러 영역에 걸치면, **해당하는 모든 영역의 문서를 각각 업데이트**
- Agent를 spawn하지 않고 직접 작업하더라도 이 규칙은 동일하게 적용

### 문서 디렉토리 구조

| 문서 경로 | 담당 영역 | 이 경로에 문서화해야 하는 작업 예시 |
|----------|----------|-------------------------------|
| `docs/dev-agent/` | Unity 클라이언트 | Scene/Prefab 변경, C# 클라이언트 스크립트, UI 구현, 애니메이션, 클라이언트 아키텍처 |
| `docs/dev-server-agent/` | C# 서버 | 서버 API 추가/변경, DB 스키마, 서버 로직, 통신 프로토콜, 서버 설정 |
| `docs/planning-agent/` | 기획 | 게임 기획서, 밸런스 설계, 시스템 기획, 콘텐츠 설계, 사양서 |
| `docs/graphics-agent/` | 그래픽 리소스 | 스프라이트, 이미지 에셋, 셰이더, 비주얼 이펙트, 리소스 가이드라인 |
| `docs/qa-agent/` | QA/테스트 | 테스트 케이스, 버그 리포트, 테스트 자동화, 품질 기준 |
| `docs/claude-code-engineer/` | Claude Code 설정 | CLAUDE.md, MCP 설정, hooks, skills, Agent 운영 규칙 |

### 판단 기준

1. **작업 결과물이 영향을 미치는 영역**을 기준으로 문서 경로를 선택한다
2. 작업 주체가 판단 기준에 포함되지 않는다
   - 예: planning-agent가 서버 API 스펙을 정리했다면 -> `docs/dev-server-agent/`에 문서화
   - 예: 사용자가 직접 Unity 스크립트를 수정했다면 -> `docs/dev-agent/`에 문서화
   - 예: dev-agent가 클라이언트-서버 연동 작업을 했다면 -> `docs/dev-agent/`와 `docs/dev-server-agent/` 모두 업데이트

### 트러블슈팅 문서 작성 (필수)

작업 중 버그, 에러, 문제 상황을 만나고 해결했을 때, 해당 영역의 `TroubleShooting/` 폴더에 **반드시** 문서를 작성한다.

- **대상**: 버그 리포트, 빌드 에러, 런타임 에러, 설정 이슈, 환경 문제 등 — 원인을 파악하고 해결한 모든 경우
- 문서 경로: `docs/{영역디렉토리}/TroubleShooting/`
- 재발 방지를 위한 자료 축적이 목적이므로, 해결 후 문서 작성을 **생략하지 않는다**

### 시스템 기획서(SystemDesign) 작성 규칙

- 게임 밸런스에 영향을 주는 **구체적 수치(데미지값, 확률, 비용, px, 색상코드 등)는 기획서에 명시 금지**
- 기획서에는 **개념과 구조만 기술**하고, 수치는 데이터 테이블을 참조하도록 안내
- 새로운 수치가 필요하면 기존 데이터 테이블에 추가하거나 새 테이블을 생성
