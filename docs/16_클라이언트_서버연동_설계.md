# 16. 클라이언트 서버 연동 설계

## 현황 분석

### 현재 아키텍처 (오프라인 모드)

```
[Screen] → GameManager.Instance.SomeMethod() → 로컬 상태 직접 변경 → SaveGame() → UI.Refresh()
```

- `GameManager`가 모든 게임 상태를 직접 보유 (Player, Tower, Dungeon, Gacha 등)
- Screen에서 `Game.Player.Talent.Upgrade()` 같이 **도메인 객체를 직접 호출**하여 상태 변경
- 변경 후 `Game.SaveGame()`으로 로컬 JSON 저장
- 네트워크 레이어 없음

### 상태 변경 지점 (Screen → GameManager)

| Screen | 액션 | 현재 구현 |
|--------|------|----------|
| TalentScreen | 재능 업그레이드 | `talent.Upgrade()` → `Resources.Spend()` 직접 호출 |
| TalentScreen | 마일스톤 수령 | `ClaimedMilestones.Add()` → `Resources.Add()` 직접 호출 |
| GachaScreen | 1회/10회 뽑기 | `Game.PullGacha()` → 결과 Equipment를 직접 인벤토리 추가 |
| EquipmentScreen | 장비 강화/장착 | `Equipment.Upgrade()`, `Slot.Equip()` 직접 호출 |
| PetScreen | 먹이/등급업 | `Pet.Feed()`, `Pet.UpgradeGrade()` 직접 호출 |
| ChapterScreen | 챕터 시작/인카운터 | `Game.StartChapter()` → 로컬 Chapter 생성 |
| ContentScreen | 던전/타워/광산 | `Game.ChallengeDungeon()` 등 직접 호출 |
| EventScreen | 퀘스트 보상 수령 | `EventManager` 직접 호출 |

---

## 목표 아키텍처 (서버 권위 모델)

```
[Screen] → ApiClient.Request() → [Server] → 응답(갱신된 상태) → GameState.Apply() → EventBus.Publish → UI.Refresh()
```

### 핵심 원칙

1. **서버가 진실의 원천(Source of Truth)** — 클라이언트는 서버 응답으로만 상태 갱신
2. **클라이언트는 요청만** — 상태를 직접 변경하지 않고 서버에 액션 요청
3. **낙관적 업데이트(Optimistic Update)** — UX를 위해 요청 즉시 UI 반영, 서버 응답으로 확정/롤백
4. **오프라인 폴백** — 네트워크 불가 시 기존 로컬 모드로 동작 (개발/테스트용)

---

## 구현 설계

### 1. 네트워크 레이어 (Infrastructure)

#### ApiClient

```
Assets/_Project/Scripts/Infrastructure/Network/
  ApiClient.cs          — HTTP 클라이언트 (UnityWebRequest 래핑)
  ApiEndpoints.cs       — 엔드포인트 상수
  ApiResponse.cs        — 공통 응답 DTO
  NetworkMode.cs        — Online/Offline 모드 플래그
```

**ApiClient 역할:**
- JWT 토큰 관리 (로그인 후 자동 부착)
- 요청/응답 JSON 직렬화 (Newtonsoft.Json 사용, 기존과 동일)
- 네트워크 에러 → 재시도 (최대 3회, 지수 백오프)
- 타임아웃 처리 (10초)
- 오프라인 모드 시 로컬 처리 위임

#### 응답 구조

서버의 모든 상태 변경 API는 **변경된 데이터만** 응답:

```csharp
public class ApiResponse<T>
{
    public bool Success;
    public string Error;
    public T Data;
    public StateDelta Delta;
}

public class StateDelta
{
    public Dictionary<string, float> Resources;
    public PlayerDelta Player;
    public List<EquipmentData> NewEquipments;
    public List<string> RemovedEquipmentIds;
    public List<PetData> NewPets;
    public long ServerTimestamp;
}
```

### 2. GameState (Services)

`GameManager`의 상태 보유 역할을 분리:

```
Assets/_Project/Scripts/Services/
  GameState.cs          — 게임 상태 보관 + 서버 응답 적용
  GameManager.cs        — (기존) 유지, GameState 참조로 변경
```

**GameState 역할:**
- `Player`, `Tower`, `Dungeon` 등 모든 상태 보관 (기존 GameManager에서 이동)
- `ApplyDelta(StateDelta)` — 서버 응답의 변경분을 로컬 상태에 반영
- `ApplyFullSync(SaveState)` — 로그인 시 전체 상태 동기화
- 변경 후 `EventBus.Publish<StateChangedEvent>()` 발행

### 3. 서비스 레이어 리팩토링

**변경 패턴:** 직접 상태 변경 → 서버 요청 + 응답 적용

#### Before (현재)
```csharp
// TalentScreen.OnUpgradeClicked()
var result = talent.Upgrade(statType, gold);
if (result.IsOk())
{
    Game.Player.Resources.Spend(ResourceType.GOLD, result.Data.Cost);
    Game.SaveGame();
    UI.Refresh();
}
```

#### After (서버 연동)
```csharp
// TalentScreen.OnUpgradeClicked()
async void OnUpgradeClicked(StatType statType)
{
    SetButtonsInteractable(false);
    var response = await Game.Api.TalentUpgrade(statType);
    if (response.Success)
    {
        Game.State.ApplyDelta(response.Delta);
    }
    else
    {
        UI.ShowError(response.Error);
    }
    SetButtonsInteractable(true);
}
```

#### GameManager API 메서드 변환 목록

| 현재 메서드 | 서버 API | 요청 파라미터 | 응답 Delta |
|------------|----------|-------------|-----------|
| `StartChapter(id, type)` | `POST /chapter/start` | chapterId, type | Resources(스태미나 차감), Chapter 상태 |
| `PullGacha()` | `POST /gacha/pull` | chestType, count | Resources(젬 차감), NewEquipments, PityCount |
| `PullGacha10()` | `POST /gacha/pull` | chestType, count=10 | 동일 |
| `ChallengeDungeon(type)` | `POST /dungeon/challenge` | dungeonType | DungeonState, BattleSeed |
| `OnDungeonBattleResult()` | `POST /dungeon/result` | dungeonType, battleLog | Resources(보상) |
| `SweepDungeon(type)` | `POST /dungeon/sweep` | dungeonType | Resources(보상), DungeonCount |
| `ClaimChapterTreasure(id)` | `POST /treasure/claim` | milestoneId | Resources(보상), ClaimedMilestones |
| `ClaimAttendance()` | `POST /attendance/claim` | - | CheckedDays, Resources/Pet/Equipment |
| `CheckDailyReset()` | 로그인 시 서버 자동 처리 | - | - |

### 4. UI 상태 반영 패턴

#### EventBus 기반 자동 갱신

```csharp
public struct ResourcesChangedEvent { }
public struct EquipmentChangedEvent { }
public struct PlayerStatsChangedEvent { }
public struct ChapterStateChangedEvent { }
```

`GameState.ApplyDelta()` 실행 시 변경된 영역에 해당하는 이벤트 발행.
각 Screen은 관심 있는 이벤트만 구독하여 `Refresh()` 호출.

#### 로딩 상태 처리

서버 요청 중 버튼 비활성화 + 로딩 인디케이터 표시.
이중 요청 방지를 위한 요청 잠금.

### 5. 오프라인/온라인 모드 전환

```csharp
public enum NetworkMode
{
    ONLINE,
    OFFLINE
}
```

- `OFFLINE`: 기존 로직 그대로 (로컬 도메인 호출 + SaveGame)
- `ONLINE`: 서버 API 호출 + Delta 적용

`GameManager`가 현재 모드에 따라 분기:

```csharp
public async Task<Result<UpgradeResult>> TalentUpgrade(StatType stat)
{
    if (_networkMode == NetworkMode.OFFLINE)
        return TalentUpgradeLocal(stat);
    return await TalentUpgradeOnline(stat);
}
```

개발/테스트 시 오프라인 모드 사용 → 서버 배포 후 온라인 모드 전환.

### 6. 전투 시스템 연동

전투는 특수 케이스:
- **전투 시작**: 서버에서 시드 + 적 데이터 수신
- **전투 진행**: 클라이언트 로컬에서 실행 (연출/애니메이션)
- **전투 종료**: 서버에 결과(배틀 로그) 전송 → 서버 재검증

```
1. Client → POST /battle/start → Server
2. Server: 시드 생성, 적 생성, 세션 저장 → {seed, enemy, sessionId}
3. Client: 로컬 전투 시뮬레이션 (서버 시드 사용)
4. Client → POST /battle/result {sessionId, log} → Server
5. Server: 동일 시드로 재시뮬레이션, 결과 비교
6. Server → {rewards, delta} 또는 {error: "mismatch"}
```

### 7. 세이브 시스템 변경

| 항목 | 오프라인 | 온라인 |
|------|---------|--------|
| 세이브 위치 | 로컬 JSON | 서버 DB (진실) + 로컬 캐시 |
| 세이브 시점 | 매 액션 후 | 서버가 자동 관리 |
| 로드 시점 | 앱 시작 시 | 로그인 → `GET /sync/full` |
| 복구 | 로컬 파일 | 서버 데이터 기준 |

## 구현 순서

### Phase 1: 기반 구축
1. `ApiClient` + `ApiResponse` + `NetworkMode` 구현
2. `GameState` 클래스 분리 (GameManager에서 상태 추출)
3. `StateDelta` + `ApplyDelta()` 구현
4. 오프라인 모드에서 기존 동작 유지 확인

### Phase 2: 핵심 API 연동
5. 인증 (로그인 → JWT → 전체 동기화)
6. 재화 변경 API (재능 업그레이드, 가챠)
7. 장비/펫 API
8. 전투 시작/결과 API

### Phase 3: 나머지 시스템
9. 챕터/인카운터 API
10. 컨텐츠 API (타워, 던전, 광산)
11. 출석/퀘스트 API
12. 일일 리셋 서버 자동 처리

### Phase 4: 안정화
13. 에러 처리 / 재시도 / 타임아웃 UX
14. 낙관적 업데이트 + 롤백 처리
15. 오프라인 → 온라인 전환 시 동기화 충돌 해결

```
CatCatGo.Infrastructure  ← 변경 없음 + Network/ 추가
```
