# 16. 클라이언트-서버 연동 설계

---

## 1. 현황 분석

### 현재 아키텍처 (오프라인 모드)

```
[Screen] → GameManager.Instance.Method() → 로컬 상태 직접 변경 → SaveGame() → UI.Refresh()
```

- `GameManager`가 모든 게임 상태를 직접 보유 (Player, Tower, Dungeon, Gacha 등)
- Screen에서 도메인 객체를 직접 호출하여 상태 변경
- 변경 후 `Game.SaveGame()`으로 로컬 JSON 저장
- `EventBus`는 구현되어 있으나 **미사용** (Subscribe/Publish 호출 0건)

### 상태 변경 패턴 분류

#### A. GameManager 메서드 경유

Screen이 GameManager의 public 메서드를 호출하고, 메서드 내부에서 상태 변경이 일어남.

| Screen | 메서드 | 내부 동작 |
|--------|--------|----------|
| ChapterScreen | `Game.StartChapter(id, type)` | Resources.Spend(STAMINA) + Chapter 생성 |
| GachaScreen | `Game.PullGacha()` | Resources.Spend(GEMS) + TreasureChest.Pull → PullResult 반환 |
| GachaScreen | `Game.PullGacha10()` | 10회 Pull |
| ContentScreen | `Game.ChallengeDungeon(type)` | 던전 입장 횟수 차감 + 전투 생성 |
| ContentScreen | `Game.OnDungeonBattleResult(type, state)` | 보상 지급 |
| ContentScreen | `Game.SweepDungeon(type)` | 입장 횟수 차감 + 즉시 보상 |
| ChapterTreasureScreen | `Game.ClaimChapterTreasure(milestoneId)` | 마일스톤 수령 + 보상 |
| EventScreen | `Game.ClaimAttendance()` | 출석 체크인 + 보상 |

#### B. Screen이 도메인 객체 직접 변경

Screen 코드에서 GameManager 래핑 없이 도메인 엔티티의 변경 메서드를 직접 호출.

| Screen | 코드 | 변경 대상 |
|--------|------|----------|
| TalentScreen | `talent.Upgrade(statType, gold)` | Talent._atkLevel/\_hpLevel/\_defLevel |
| TalentScreen | `Player.Resources.Spend(GOLD, cost)` | Resources |
| EquipmentScreen | `equipment.Upgrade(stones)` | Equipment.Level |
| EquipmentScreen | `Player.Resources.Spend(EQUIPMENT_STONE, cost)` | Resources |
| EquipmentScreen | `Player.EquipFromInventory(id)` | Inventory ↔ EquipmentSlots |
| EquipmentScreen | `Player.UnequipToInventory(slotType, index)` | EquipmentSlots → Inventory |
| EquipmentScreen | `Player.SellEquipment(id)` | Inventory 삭제 + Resources.Add(GOLD) |
| EquipmentScreen | `ForgeService.Merge(group, Rng)` + `RemoveFromInventory` + `AddToInventory` | Inventory |
| PetScreen | `pet.Feed(foodAmount)` | Pet.Exp, Pet.Level |
| PetScreen | `Player.Resources.Spend(PET_FOOD, n)` | Resources |
| PetScreen | `Player.Resources.Spend(PET_EGG, 1)` + `Player.AddPet(pet)` | Resources + OwnedPets |
| PetScreen | `Player.SetActivePet(pet)` | ActivePet |
| QuestScreen | `evt.ClaimMissionReward(missionId)` + `Resources.Add()` | Mission.Claimed + Resources |
| ContentScreen | `tower.Challenge(...)` + `Resources.Spend(CHALLENGE_TOKEN)` | Tower + Resources |
| ContentScreen | `miner.Mine(pickaxes)` + `Resources.Spend(PICKAXE)` | GoblinMiner + Resources |
| ContentScreen | `miner.UseCart(Rng)` + `Resources.Add()` | GoblinMiner + Resources |
| ContentScreen | `catacomb.StartRun()` / `OnBattleResult()` / `EndRun()` | Catacomb |
| ChapterScreen | `Player.Resources.Add(type, amt)` (인카운터 보상) | Resources |
| ChapterScreen | `Player.UpdateBestSurvivalDay(...)` | BestSurvivalDays |

#### C. 필드 직접 변경 (래핑 없음)

| Screen | 코드 | 변경 대상 |
|--------|------|----------|
| TalentScreen | `Player.ClaimedMilestones.Add(key)` | HashSet 직접 변경 |
| ChapterScreen | `Player.ClearedChapterMax = Max(...)` | int 필드 직접 대입 |
| ChapterScreen | `Game.CurrentChapter = null` | 챕터 세션 종료 |

#### D. GachaScreen 분할 책임 문제

`Game.PullGacha()`가 젬 차감 + 풀링을 수행하지만, 결과 장비/보상을 인벤토리/리소스에 추가하는 것은 **Screen 코드에서 별도 수행**. 서버 연동 시 이 분할을 하나의 원자적 API로 통합해야 함.

## 2. 목표 아키텍처

```
[Screen] → Game.SomeMethod() → (OFFLINE: 로컬 처리 / ONLINE: API 호출) → GameState.ApplyDelta() → EventBus → UI.Refresh()
```

### 핵심 원칙

1. **서버가 진실의 원천(Source of Truth)** — 온라인 모드에서 클라이언트는 서버 응답으로만 상태 갱신
2. **Screen은 NetworkMode를 모름** — GameManager가 내부에서 분기
3. **오프라인 폴백** — 네트워크 불가 시 기존 로컬 모드로 동작

## 3. StateDelta 스키마

서버 응답에 포함되는 상태 변경분. null 필드 = 변경 없음.

```csharp
public class StateDelta
{
    public Dictionary<string, float> Resources;

    public TalentDelta Talent;
    public HeritageDelta Heritage;

    public List<EquipmentData> AddedEquipments;
    public List<string> RemovedEquipmentIds;
    public List<EquipmentUpgradeDelta> UpgradedEquipments;
    public List<EquipSlotDelta> EquipmentSlotChanges;

    public List<PetData> AddedPets;
    public List<PetUpdateDelta> UpdatedPets;
    public string ActivePetId;

    public int? ClearedChapterMax;
    public Dictionary<string, int> BestSurvivalDays;
    public List<string> AddedClaimedMilestones;

    public TowerDelta Tower;
    public CatacombDelta Catacomb;
    public DungeonDelta Dungeons;
    public int? GoblinOreCount;
    public int? PityCount;
    public List<string> AddedCollectionIds;
    public List<MissionDelta> MissionUpdates;
    public AttendanceDelta Attendance;
    public ChapterSessionDelta ChapterSession;

    public long ServerTimestamp;
}
```

## 4. API 엔드포인트 전체 명세

### 공통 응답 형식

```csharp

{
```

### 4.1 인증

#### POST /api/auth/register

```
Request:  { deviceId: string }
Response: { accessToken: string, refreshToken: string, accountId: string, isNewAccount: bool }
검증: deviceId 형식, 중복 체크
```

#### POST /api/auth/login

```
Request:  { deviceId: string }
Response: { accessToken: string, refreshToken: string, accountId: string }
검증: 계정 존재 여부
```

#### POST /api/auth/refresh

```
Request:  { refreshToken: string }
Response: { accessToken: string, refreshToken: string }
검증: 토큰 유효성 + 만료
```

#### POST /api/auth/link

```
Request:  { socialType: "GOOGLE"|"APPLE", socialToken: string }
Response: { success: bool }
검증: 소셜 토큰 검증, 중복 연동 체크
```

```
Response: { saveState: SaveState, serverTimestamp: long }
```

### 4.7 챕터 (멀티스텝 세션)

챕터는 서버에 Redis 세션을 생성하고 여러 API 호출에 걸쳐 진행됨.

#### POST /api/chapter/start

```
Request:  { chapterId: int, chapterType: "SIXTY_DAY" }
Response: { delta: StateDelta, sessionId: string, seed: int }
검증: chapterId == ClearedChapterMax + 1, STAMINA ≥ 5, 활성 세션 없음
서버: Redis에 ChapterSession 생성 { seed, playerStatsSnapshot, RNG 상태 }
Delta: Resources(STAMINA↓), ChapterSession(새 세션)
```

#### POST /api/chapter/advance-day

```
Request:  { sessionId: string }
Response: {
    encounter: EncounterData | null,
    battleRequired: "NONE"|"COMBAT"|"ELITE"|"MIDBOSS"|"BOSS",
    battleSeed: int | null,
    enemyTemplateId: string | null,
    delta: StateDelta
}
검증: 세션 존재, 진행 중
서버: Chapter.AdvanceDay() 실행 (저장된 시드 사용)
Delta: ChapterSession(currentDay, 카운터)
```

#### POST /api/chapter/resolve-encounter

```
Request:  { sessionId: string, choiceIndex: int }
Response: { delta: StateDelta }
검증: 대기 중인 인카운터 존재, choiceIndex 유효
서버: Chapter.ResolveEncounter() 실행, goldMultiplier 서버측 적용
Delta: Resources(인카운터 보상), ChapterSession(hp, gold, skills, 카운터)
```

#### POST /api/chapter/reroll

```
Request:  { sessionId: string }
Response: { encounter: EncounterData, delta: StateDelta }
검증: rerollsRemaining > 0
Delta: ChapterSession(rerollsRemaining↓)
```

#### POST /api/chapter/select-skill

```
Request:  { sessionId: string, skillId: string }
Response: { delta: StateDelta }
검증: skillId가 서버 생성 후보 목록에 존재
Delta: ChapterSession(sessionSkills, maxHp 재계산)
```

#### POST /api/chapter/battle-result

```
Request:  {
    sessionId: string,
    battleSeed: int,
    result: "VICTORY"|"DEFEAT",
    turnCount: int,
    playerRemainingHp: int
}
Response: { delta: StateDelta, verified: bool, goldEarned: int }
검증: 시드 일치, 서버 리플레이로 결과 검증, 턴 수 허용 범위(±1)
Delta:
  - 일반 전투 승리: Resources(전투 골드), ChapterSession(hp)
  - 보스 승리: Resources(클리어 골드/젬), ClearedChapterMax, BestSurvivalDays, ChapterSession(종료)
  - 패배: BestSurvivalDays, ChapterSession(종료)
```

#### POST /api/chapter/abandon

```
Request:  { sessionId: string }

Delta: BestSurvivalDays, ChapterSession(종료)
```

## 5. 인증 흐름

```
앱 시작
  │
  ├─ 저장된 JWT 있음?
  │   YES → POST /auth/refresh { refreshToken }
  │          ├─ 성공 → GET /sync/full → GameState 적용 → ONLINE 모드
  │          └─ 실패(만료) → POST /auth/login { deviceId } → 토큰 → /sync/full
  │
  │   NO → 저장된 deviceId 있음?
  │         YES → POST /auth/login { deviceId } → 토큰 → /sync/full
  │         NO  → POST /auth/register { newDeviceId } → 토큰 → /sync/full (신규)
  │
  └─ 네트워크 불가?
       → OFFLINE 모드 → 로컬 세이브 로드
```

JWT 수명:
- Access Token: 1시간
- Refresh Token: 30일
- 클라이언트: PlayerPrefs에 암호화 저장
- ApiClient: 요청마다 만료 체크, 5분 미만 시 자동 갱신
- 401 응답 시: refresh → 실패면 재로그인 → 실패면 OFFLINE

## 6. 전투 검증 시스템

### 6.1 챕터 전투 (클라이언트 실행 + 서버 검증)

챕터 전투는 애니메이션 연출이 있으므로 클라이언트에서 실행하고 서버에서 시드 기반 리플레이로 검증.

```
1. Client → POST /chapter/advance-day
2. Server: 시드 생성, 플레이어 스냅샷 저장
           → { battleSeed, enemyTemplateId }
3. Client: SeededRandom(battleSeed)로 Battle 생성 + 애니메이션 연출
4. Client → POST /chapter/battle-result { seed, result, turnCount, playerRemainingHp }
5. Server: 동일 시드 + 스냅샷으로 Battle.RunToCompletion(50)
           결과 비교: result 일치, turnCount ±1 이내
6. Server → 일치: { delta(보상), verified: true }
           불일치: { verified: false, cheat_flag 기록 }
```

### 6.2 컨텐츠 전투 (서버 실행)

타워/던전/카타콤은 자동 전투(ContentScreen에서 `RunToCompletion` 후 결과만 표시).
서버에서 전투를 직접 실행하고 결과만 응답. 시드 검증 불필요.

```
1. Client → POST /content/tower/challenge
2. Server: 전투 생성 + RunToCompletion(50) + 보상 계산
3. Server → { battleResult, reward, delta }
```

## 7. 챕터 세션 관리

챕터는 게임 내 가장 복잡한 멀티스텝 플로우. 서버에 Redis 세션으로 관리.

### Redis 세션 구조

키: `chapter:{accountId}` (계정당 1개)

```
{
    sessionId: string,
    chapterId: int,
    seed: int,
    rngState: int,
    playerStatsSnapshot: Stats,
    playerSkillIds: string[],
    currentDay: int,
    sessionCurrentHp: int,
    sessionMaxHp: int,
    sessionGold: int,
    jungbakCount: int,
    daebakCount: int,
    rerollsRemaining: int,
    pendingEncounter: EncounterData | null,
    pendingBattleSeed: int | null,
    pendingEnemyTemplateId: string | null,
    status: "IN_PROGRESS" | "ENDED"
}
```

### 플로우

```
Phase 1: START
  POST /chapter/start → 세션 생성, STAMINA 차감

Phase 2: ENCOUNTER LOOP (매 day 반복)
  POST /chapter/advance-day → 인카운터 또는 전투 결정
    ├─ 인카운터 → UI 표시 → POST /chapter/resolve-encounter { choiceIndex }
    ├─ 스킬 선택 → UI 표시 → POST /chapter/select-skill { skillId }
    ├─ 리롤 → POST /chapter/reroll
    └─ 전투 → 클라이언트 실행 → POST /chapter/battle-result
         └─ 엘리트/미드보스 승리 → 스킬 후보 3개 전달 → POST /chapter/select-skill

Phase 3: BOSS (day 60)
  POST /chapter/advance-day → battleRequired: "BOSS"
  클라이언트 보스 전투 실행
  POST /chapter/battle-result
    ├─ 승리: ClearedChapterMax↑, 클리어 보상, 세션 종료
    └─ 패배: BestSurvivalDays 갱신, 세션 종료

Phase 4: ABANDON
  POST /chapter/abandon → BestSurvivalDays 갱신, 세션 삭제
```

## 8. 오프라인/온라인 모드

### GameManager 듀얼 모드 패턴

```csharp
public enum NetworkMode { ONLINE, OFFLINE }
```

GameManager의 모든 public 메서드가 내부에서 모드 분기:

```csharp
public async Task<Result<T>> SomeAction(params)
{
    if (_networkMode == NetworkMode.OFFLINE)
        return SomeActionLocal(params);      // 기존 로직 그대로
    return await SomeActionOnline(params);   // API 호출 + Delta 적용
}
```

**Screen은 모드를 모름.** GameManager 메서드만 호출하면 됨.
OFFLINE 모드에서 async 메서드는 완료된 Task를 즉시 반환 (동기처럼 동작).

### 모드 전환 시나리오

| 시나리오 | 처리 |
|---------|------|
| 앱 시작, 서버 접속 성공 | ONLINE 모드 |
| 앱 시작, 서버 접속 실패 | OFFLINE 모드 + 로컬 세이브 |
| ONLINE 중 API 3회 연속 실패 | OFFLINE 전환 + 현재 상태 로컬 저장 + 토스트 알림 |
| OFFLINE 중 재연결 | POST /sync/push(로컬→서버) → 성공 시 GET /sync/full → ONLINE 전환 |
| 동기화 충돌 | 서버 데이터 우선 (로컬 진행분 폐기) |

모든 Screen의 상태 변경 메서드를 async로 전환:

```csharp
private bool _isRequestPending;

private async void OnSomeAction()
{
    if (_isRequestPending) return;
    _isRequestPending = true;
    SetInteractable(false);

    var result = await Game.SomeApiCall(params);
    if (result.IsOk())
        UI.Refresh();
    else
        UI.ShowToast(result.Message);

    _isRequestPending = false;
    SetInteractable(true);
}
```
