# 서버/클라 연동 테스트 흐름

## 작성일
2026-03-07

## 기획 의도
서버/클라이언트 연동 구현의 정합성을 검증하기 위한 테스트 흐름 정의.
ONLINE/OFFLINE 모드별 데이터 흐름이 올바르게 동작하는지 확인한다.

## 목표
- 모든 GameManager async 메서드의 ONLINE/OFFLINE 분기 정상 동작 확인
- Screen에서 직접 상태 변경(B/C 패턴) 잔존 여부 점검
- 계정 관리(리셋/삭제) 흐름의 서버-로컬 동기화 검증
- StateDelta → ApplyDelta → EventBus → UI 갱신 파이프라인 검증

## 관련 문서
- docs/dev-agent/SystemDesign/클라이언트_서버연동_설계_상세.md

## 1. 데이터 흐름 원칙

```
[ONLINE 모드]
Screen → GameManager.XxxAsync(callback)
  → API 호출 → 서버 응답(StateDelta)
  → ApplyServerDelta → GameState.ApplyDelta
  → EventBus.Publish → Screen 자동 갱신

[OFFLINE 모드]
Screen → GameManager.XxxAsync(callback)
  → 로컬 로직 직접 실행 → callback 반환
  → Screen에서 결과 처리 + UI.Refresh
```

**핵심 규칙**: ONLINE 모드에서 Screen은 상태를 직접 변경하면 안 된다. 서버 Delta가 유일한 상태 변경 소스.

## 4. EventBus 파이프라인 검증

```
ApplyDelta 변경 → EventBus 발행 → Screen 구독 → UI 자동 갱신
```

| Delta 필드 | 발행 이벤트 | 구독 Screen |
|-----------|-----------|------------|
| Resources | ResourcesChangedEvent | TalentScreen, EquipmentScreen, PetScreen, ContentScreen, ChapterScreen |
| Talent | TalentChangedEvent | TalentScreen |
| Heritage | HeritageChangedEvent | TalentScreen |
| AddedEquipments/UpgradedEquipments/EquipmentSlotChanges | EquipmentChangedEvent | EquipmentScreen |
| RemovedEquipmentIds/AddedEquipments (인벤토리) | InventoryChangedEvent | EquipmentScreen |
| AddedPets/UpdatedPets/ActivePetId | PetChangedEvent | PetScreen |
| Tower | TowerChangedEvent | ContentScreen |
| Catacomb | CatacombChangedEvent | ContentScreen |
| Dungeons | DungeonChangedEvent | ContentScreen |
| MissionUpdates | QuestChangedEvent | QuestScreen |
| Attendance | AttendanceChangedEvent | EventScreen |
| ChapterSession | ChapterStateChangedEvent | ChapterScreen |
| (스탯 관련 변경) | PlayerStatsChangedEvent | MainScreen |
