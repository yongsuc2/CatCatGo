# 클라이언트 서버연동 설계

## 작성일
2026-03-07

## 기획 의도
현재 클라이언트는 오프라인 모드로 동작하며, Screen에서 도메인 객체를 직접 변경하고 로컬 JSON으로 세이브한다. 서버 연동 시 이 구조를 유지하면서 ONLINE/OFFLINE 듀얼 모드를 지원하고, Screen은 NetworkMode를 모르게 GameManager가 내부에서 분기하도록 설계한다.

## 목표
- Screen 코드 변경 최소화 (GameManager 메서드만 호출)
- StateDelta 기반 상태 갱신으로 서버 응답을 원자적으로 적용
- EventBus를 통한 UI 자동 갱신
- 오프라인 폴백 지원

## 동작 방식

### 목표 아키텍처

```
[Screen] -> Game.SomeMethod() -> (OFFLINE: 로컬 / ONLINE: API) -> GameState.ApplyDelta() -> EventBus -> UI.Refresh()
```

### 현황 분석

현재 상태 변경 패턴 3가지:
- A. GameManager 메서드 경유 (StartChapter, PullGacha 등)
- B. Screen이 도메인 객체 직접 변경 (talent.Upgrade, equipment.Upgrade 등)
- C. 필드 직접 변경 (ClaimedMilestones.Add, ClearedChapterMax = Max 등)

서버 연동 시 B, C 패턴을 모두 A 패턴(GameManager 메서드)으로 통합해야 함.

### StateDelta 스키마

서버 응답에 포함되는 상태 변경분. null 필드 = 변경 없음.

주요 Delta 필드:
- Resources: 재화 변동
- Talent/Heritage: 성장 시스템 변동
- AddedEquipments/RemovedEquipmentIds: 장비 인벤토리 변동
- AddedPets/UpdatedPets/ActivePetId: 펫 변동
- ClearedChapterMax/BestSurvivalDays: 진행도 변동
- Tower/Catacomb/Dungeons: 컨텐츠 변동
- ChapterSession: 챕터 세션 변동

### API 설계 (30개 엔드포인트)

| 카테고리 | 엔드포인트 수 | 주요 API |
|---------|-------------|----------|
| 인증 | 4 | register, login, refresh, link |
| 동기화 | 2 | sync/full, sync/push |
| 재능 | 3 | upgrade, claim-milestone, claim-all |
| 장비 | 6 | upgrade, equip, unequip, sell, forge, bulk-forge |
| 펫 | 3 | hatch, feed, deploy |
| 가챠 | 2 | pull, pull10 |
| 챕터 | 7 | start, advance-day, resolve-encounter, reroll, select-skill, battle-result, abandon |
| 컨텐츠 | 9 | tower, dungeon, dungeon/sweep, goblin/mine, goblin/cart, catacomb/* |
| 일일/이벤트 | 3 | attendance/claim, quest/claim, quest/claim-all |
| 보물/유산 | 2 | treasure/claim, heritage/upgrade |

### 오프라인/온라인 모드

GameManager의 모든 public 메서드가 내부에서 모드 분기:
- OFFLINE: 기존 로컬 로직 그대로 실행
- ONLINE: API 호출 -> StateDelta 적용

모드 전환 시나리오:
- 앱 시작, 서버 접속 성공 -> ONLINE
- 앱 시작, 서버 접속 실패 -> OFFLINE + 로컬 세이브
- ONLINE 중 API 3회 연속 실패 -> OFFLINE 전환
- OFFLINE 중 재연결 -> sync/push -> sync/full -> ONLINE

### 구현 순서 (8 Phase)

| Phase | 내용 |
|-------|------|
| 1 | 기반 구축 (GameEvents, GameState, StateDelta) |
| 2 | 네트워크 레이어 (ApiClient, JWT) |
| 3 | Screen 리팩토링 + GameManager 듀얼 모드 |
| 4 | 인증 + 동기화 |
| 5 | 단순 시스템 API (재능/장비/펫/가챠/유산) |
| 6 | 컨텐츠 API (타워/던전/광산/카타콤/출석/퀘스트) |
| 7 | 챕터 세션 |
| 8 | 안정화 (에러 처리, 모드 전환 UX) |
