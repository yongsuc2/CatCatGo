# 17. 서버 테스트 설계

## 개요

| 항목 | 값 |
|------|-----|
| 테스트 프레임워크 | xUnit 2.x |
| Mock 라이브러리 | NSubstitute 5.x |
| 대상 | Server.Core 서비스 레이어 (5개 서비스) |
| 테스트 유형 | 단위 테스트 (Unit Test) |
| 실행 환경 | .NET 8.0, CI/로컬 동일 |

## 테스트 범위

### 대상 서비스 및 테스트 수

| 서비스 | 테스트 파일 | 테스트 케이스 수 | 핵심 검증 |
|--------|-------------|-----------------|-----------|
| AuthService | AuthServiceTests.cs | 8 | 회원가입, 로그인, 차단 계정, JWT 토큰, 리프레시 토큰 |
| SaveService | SaveServiceTests.cs | 6 | 신규 저장, 클라이언트 최신, 서버 최신, 로드, 체크섬 |
| ShopService | ShopServiceTests.cs | 7 | 카탈로그 필터링, 구매 성공, 중복 영수증, 비활성 상품, 미지원 스토어, 영수증 검증 실패 |
| BattleVerifier | BattleVerifierTests.cs | 7 | 전투 시작, 정상 보고, 잘못된 ID, 계정 불일치, 중복 완료, 시드 불일치, 속도핵 |
| ArenaService | ArenaServiceTests.cs | 7 | 신규 매칭, 기존 매칭, 자기 제외, 승리/무승부/패배 포인트, 티어 계산 |

**총 테스트 케이스: 35개**

## 테스트 전략

### Mock 기반 단위 테스트
- 모든 Repository 인터페이스를 NSubstitute로 Mock
- 외부 의존성(DB, Redis, 스토어 API) 없이 서비스 로직만 검증
- IReceiptVerifier Mock으로 결제 검증 시나리오 커버

### 테스트 명명 규칙
```
[메서드명]_[시나리오]_[기대결과]
```
예: `RegisterAsync_NewDevice_CreatesAccountAndReturnsTokens`

### 서비스별 검증 포인트

#### AuthService (8 케이스)
| # | 테스트 | 검증 |
|---|--------|------|
| 1 | 신규 디바이스 등록 | Account 생성, JWT 발급, IsNewAccount=true |
| 2 | 기존 디바이스 등록 | 기존 계정 반환, IsNewAccount=false, Account 미생성 |
| 3 | 디바이스 로그인 성공 | JWT 발급, LastLoginAt 갱신, RefreshToken 갱신 |
| 4 | 미등록 디바이스 로그인 | null 반환 |
| 5 | 차단 계정 로그인 | null 반환 |
| 6 | JWT 토큰 구조 | 유효한 JWT, 올바른 Claims(AccountId, DisplayName) |
| 7 | DisplayName 기본값 | DisplayName 미지정 시 "Player_" 접두어 자동 생성 |
| 8 | RefreshAsync 미구현 | null 반환 |

#### SaveService (6 케이스)
| # | 테스트 | 검증 |
|---|--------|------|
| 1 | 서버에 세이브 없음 → 신규 저장 | Action="ACCEPTED", 데이터 저장 |
| 2 | 클라이언트가 최신 | Action="ACCEPTED", 서버 데이터 갱신 |
| 3 | 서버가 최신 | Action="SERVER_NEWER", 서버 데이터 반환 |
| 4 | 로드 성공 | Action="LOADED", 데이터/버전 반환 |
| 5 | 로드 실패 (세이브 없음) | null 반환 |
| 6 | 체크섬 생성 | SHA256 해시가 올바르게 계산 |

#### ShopService (7 케이스)
| # | 테스트 | 검증 |
|---|--------|------|
| 1 | 전체 카탈로그 | 활성 상품만 반환 |
