# 카피바라고! 게임 서버 아키텍처

## 개요

| 항목 | 값 |
|------|-----|
| 서버 프레임워크 | ASP.NET Core 8.0 |
| 데이터베이스 | PostgreSQL 16 |
| 캐시/세션 | Redis 7 |
| 인증 | JWT + Refresh Token |
| 프로토콜 | REST API (HTTPS) |
| 컨테이너 | Docker Compose |

## 서버가 필요한 이유

| 기능 | 클라이언트 단독 불가 이유 |
|------|--------------------------|
| 결제(IAP) | 영수증 위변조 방지 — Google Play / App Store 서버 검증 필수 |
| 상품 관리 | 이벤트 패키지, 가격 변경을 앱 업데이트 없이 제어 |
| 계정 관리 | 기기 변경, 클라우드 세이브, 계정 복구 |
| 치트 검증 | SeededRandom 기반 전투 재현 → 결과 불일치 탐지 |
| 비동기 PvP | 아레나 매칭/랭킹 중앙 관리 |
| 일일 리셋 | 서버 시간 기준 통일 (클라이언트 시간 조작 방지) |

## 핵심 설계 원칙

### 도메인 로직 공유
```
Unity Client                    Game Server
┌───────────────┐              ┌───────────────┐
│ Presentation  │              │  Server.Api   │
│ Services      │              │  Server.Core  │
│ ┌───────────┐ │              │ ┌───────────┐ │
│ │  Domain   │◄├──── 동일 ────►│ │  Domain   │ │
│ │  Infra    │ │   어셈블리    │ │  Infra    │ │
│ └───────────┘ │              │ └───────────┘ │
└───────────────┘              └───────────────┘
```

- Domain, Infrastructure 어셈블리는 Unity 비의존 → 서버에서 그대로 참조
- 전투 공식, 스탯 계산, 밸런스 데이터를 서버가 동일하게 실행
- SeededRandom으로 전투 결과를 서버에서 독립 재현 → 치트 검증

### 서버 권위 모델 (Server-Authoritative)

| 영역 | 권위 | 설명 |
|------|------|------|
| 재화 변동 | 서버 | 골드/보석/스태미나 증감은 서버만 가능 |
| 결제 | 서버 | 스토어 영수증 서버 검증 후 지급 |
| 세이브 | 서버 | 서버 DB가 정본, 클라이언트는 캐시 |
| 전투 결과 | 클라이언트 주도 + 서버 검증 | 클라이언트가 결과 보고 → 서버가 시드로 재현 검증 |
| 아레나 | 서버 | 매칭/랭킹/보상 서버 관리 |
| 일일 리셋 | 서버 | UTC 기준 서버 시간 사용 |
| 가챠 | 서버 | 확률 조작 방지, 천장 카운터 서버 관리 |

## API 설계

### 1. 인증 (Auth)

| Method | Endpoint | 설명 |
|--------|----------|------|
| POST | `/api/auth/register` | 게스트 계정 생성 (디바이스 ID 기반) |
| POST | `/api/auth/login` | 로그인 → JWT 발급 |
| POST | `/api/auth/refresh` | Refresh Token으로 Access Token 갱신 |
| POST | `/api/auth/link` | 소셜 계정 연동 (Google/Apple) |
| POST | `/api/auth/recover` | 계정 복구 |

#### 인증 흐름
```
[첫 실행]
Client → POST /auth/register { deviceId }
Server → 게스트 계정 생성 → { accessToken, refreshToken, accountId }

[이후 실행]
Client → POST /auth/login { deviceId }  또는  { socialToken }
Server → JWT 발급

[토큰 만료]
Client → POST /auth/refresh { refreshToken }
Server → 새 accessToken 발급
```

### 2. 세이브 (Save)

| Method | Endpoint | 설명 |
|--------|----------|------|
| GET | `/api/save` | 서버 세이브 로드 |
| PUT | `/api/save` | 서버 세이브 저장 |
| POST | `/api/save/sync` | 클라이언트-서버 세이브 동기화 (충돌 해소) |

#### 세이브 동기화 전략
```
Client Save (timestamp: T1) vs Server Save (timestamp: T2)

if T1 == T2 → 동일, 스킵
if T1 > T2  → 클라이언트가 최신, 서버 업데이트 (검증 후)
if T1 < T2  → 서버가 최신, 클라이언트에 반환
if 충돌      → 서버 우선 (서버가 정본)
```

### 3. 상점 (Shop)

| Method | Endpoint | 설명 |
|--------|----------|------|
| GET | `/api/shop/catalog` | 상품 목록 (가격, 이벤트 패키지 포함) |
| POST | `/api/shop/purchase` | 결제 영수증 제출 → 검증 → 보상 지급 |
| GET | `/api/shop/history` | 구매 이력 |
| GET | `/api/shop/subscription` | 월정액 상태 확인 |

#### 결제 흐름
```
1. Client → GET /shop/catalog → 상품 목록 수신
2. Client → 스토어 결제 진행 (Google Play / App Store)
3. Client → POST /shop/purchase { receipt, productId, store }
4. Server → 스토어 서버에 영수증 검증 요청
5. Server → 검증 성공 → DB에 구매 기록 + 보상 지급 + 세이브 업데이트
6. Server → { success, rewards }
7. Client → 보상 반영
```

### 4. 전투 검증 (Battle)

| Method | Endpoint | 설명 |
|--------|----------|------|
| POST | `/api/battle/report` | 전투 결과 제출 (시드 + 행동 로그) |
| GET | `/api/battle/verify/{id}` | 검증 결과 조회 |

#### 치트 검증 흐름
```
1. Client: 전투 시작 시 서버에서 시드 발급
   POST /api/battle/start { chapterId, day, encounterType }
   → { battleId, seed }

2. Client: 전투 수행 (로컬)

3. Client: 전투 결과 보고
   POST /api/battle/report {
     battleId,
     seed,
     result: WIN/LOSE,
     turns: 12,
     playerSkillIds: [...],
     enemyTemplateId: "ant_worker",
     rewards: { gold: 150 }
   }

4. Server: 동일 시드로 전투 재현
   - SeededRandom(seed) + 동일 스킬/적 데이터로 Battle 실행
   - 결과 비교: 턴 수, 승패, 보상량
   - 불일치 → 의심 플래그 + 보상 미지급
   - 일치 → 보상 확정 + 세이브 반영
```

### 5. 아레나 (Arena)

| Method | Endpoint | 설명 |
|--------|----------|------|
| POST | `/api/arena/match` | 매칭 요청 → 같은 티어 4명 선택 |
| POST | `/api/arena/result` | 결과 제출 (서버 검증 후 랭킹 반영) |
| GET | `/api/arena/ranking` | 티어별 랭킹 조회 |
| GET | `/api/arena/season` | 시즌 정보 |

### 6. 가챠 (Gacha)

| Method | Endpoint | 설명 |
|--------|----------|------|
| POST | `/api/gacha/pull` | 1회 뽑기 (서버에서 실행) |
| POST | `/api/gacha/pull10` | 10회 뽑기 (서버에서 실행) |
| GET | `/api/gacha/pity` | 천장 카운터 조회 |

## 데이터베이스 스키마

### accounts
```sql
CREATE TABLE accounts (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    device_id     VARCHAR(256) UNIQUE,
    social_type   VARCHAR(20),    -- GOOGLE, APPLE, NULL(게스트)
    social_id     VARCHAR(256),
    display_name  VARCHAR(50),
    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_login_at TIMESTAMPTZ,
    is_banned     BOOLEAN NOT NULL DEFAULT FALSE,
    ban_reason    TEXT
);
```

### save_data
```sql
CREATE TABLE save_data (
    account_id    UUID PRIMARY KEY REFERENCES accounts(id),
    data          JSONB NOT NULL,
    version       INT NOT NULL DEFAULT 1,
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    checksum      VARCHAR(64) NOT NULL
);
```

### products
```sql
CREATE TABLE products (
    id            VARCHAR(50) PRIMARY KEY,
    name          VARCHAR(100) NOT NULL,
    type          VARCHAR(20) NOT NULL,  -- GEMS, MONTHLY_PASS, BUNDLE, EVENT
    price_tier    VARCHAR(20),
    gems_amount   INT DEFAULT 0,
    bonus_gems    INT DEFAULT 0,
    rewards       JSONB,
    is_active     BOOLEAN NOT NULL DEFAULT TRUE,
    event_id      VARCHAR(50),
    start_at      TIMESTAMPTZ,
    end_at        TIMESTAMPTZ
);
```

### purchases
```sql
CREATE TABLE purchases (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    account_id      UUID NOT NULL REFERENCES accounts(id),
    product_id      VARCHAR(50) NOT NULL REFERENCES products(id),
    store           VARCHAR(20) NOT NULL,  -- GOOGLE_PLAY, APP_STORE
    receipt_id      VARCHAR(256) NOT NULL UNIQUE,
    receipt_data    TEXT NOT NULL,
    status          VARCHAR(20) NOT NULL,  -- PENDING, VERIFIED, FAILED, REFUNDED
    amount_paid     DECIMAL(10,2),
    currency        VARCHAR(3),
    purchased_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    verified_at     TIMESTAMPTZ
);
```

### arena_rankings
```sql
CREATE TABLE arena_rankings (
    account_id    UUID PRIMARY KEY REFERENCES accounts(id),
    tier          VARCHAR(20) NOT NULL DEFAULT 'BRONZE',
    points        INT NOT NULL DEFAULT 0,
    season        INT NOT NULL DEFAULT 1,
    wins          INT NOT NULL DEFAULT 0,
    losses        INT NOT NULL DEFAULT 0,
    player_data   JSONB NOT NULL,    -- 스냅샷 (스탯/장비/패시브)
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### cheat_flags
```sql
CREATE TABLE cheat_flags (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    account_id    UUID NOT NULL REFERENCES accounts(id),
    type          VARCHAR(50) NOT NULL,  -- BATTLE_MISMATCH, RESOURCE_ANOMALY, SPEED_HACK
    severity      INT NOT NULL DEFAULT 1,
    details       JSONB,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

## 치트 검증 상세

### 검증 대상

| 검증 항목 | 방법 | 자동 처리 |
|-----------|------|-----------|
| 전투 결과 조작 | 시드 기반 재현 → 결과 비교 | 불일치 시 보상 미지급 |
| 재화 조작 | 서버가 재화 정본 관리 | 클라이언트 값 무시 |
| 속도핵 | 전투 소요 시간 vs 턴 수 비교 | 비정상 속도 플래그 |
| 가챠 조작 | 서버에서 뽑기 실행 | 클라이언트 뽑기 불가 |
| 일일 리셋 악용 | 서버 UTC 시간 기준 | 클라이언트 시간 무시 |
| 영수증 재사용 | receipt_id UNIQUE 제약 | 중복 구매 차단 |

### BattleVerifier 재현 로직
```
Input: seed, playerStats, playerSkillIds, enemyTemplateId, chapterId, day
1. SeededRandom rng = new(seed)
2. Player BattleUnit 구성 (ComputeStats + 스킬 로드)
3. Enemy BattleUnit 구성 (EnemyTable.GetScaledStats)
4. Battle 실행 (최대 MAX_TURNS)
5. Output: { state, turnCount, totalDamage, rewards }
6. 클라이언트 보고와 비교
```
